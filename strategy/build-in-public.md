# Build in Public

## 什么是 Build in Public

"建在公开"不是"把手册公开"或"开源代码"。手册公开和开源是结果，不是策略。

Build in Public 是指在建设过程中，实时分享：

- 当前在做什么、为什么做
- 遇到了什么问题、怎么解决的
- 哪些尝试失败了、学到了什么
- 真实的数据——包括不好看的数据

GitLab 和 PostHog 是这一策略的代表。GitLab 把产品路线图和公司手册全部公开，任何决策的过程和理由都可在 handbook 中溯源。PostHog 实时公开 ARR、增长数据、定价调整的决策逻辑。YC 参投时看过他们的数据才放心，但那些数据早就被所有竞争对手看到了。

## 为什么 Build in Public 是护城河

传统直觉：公开信息会削弱护城河。

实际机制：**当信息本身不再是稀缺品时，它从资产变成了基础设施。而真正的护城河，从"怎么藏着信息"变成了"用什么速度消化这些信息"。**

可以画一条曲线：

```
信息开放程度
    ↑
    |          你在这里
    |          （GitLab/PostHog/量潮）
    |
    |      传统信任型公司
    |      （公开年报，隐藏过程）
    |
    |  传统保密型公司
    |  （配方/策略/数据全部保密）
    +------------------------→ 护城河强度
```

量潮和组织内的大量文档、战略、日记、数据全部公开，但我们不靠信息差建立护城河。我们的护城河来自**更快的闭环速度**。

## Build in Public 的通用闭环

所有业务的运转都可以抽象为一个核心循环：

```
接收 → 处理 → 交付 → 记录 → 接收...
  ↑                    ↑
 公开输入规范          公开输出标准
  ↓                    ↓
 让对方验证           比对手更快归档
```

每一次循环都是一次信息结构化的机会。结果越公开：

1. **客户越容易判断是否与我们合作** — 降低销售成本
2. **执行方越清楚要求的交付水平** — 提高匹配效率
3. **我们自己越早发现流程损耗** — journal 里说的"内部数据采集不到位"，公开后外部视角自动补位

这个循环对于任何业务都成立，区别只在于接收什么、处理什么、交付什么：

| 业务 | 接收（输入） | 处理 | 交付（输出） |
|------|------------|------|------------|
| 数据服务 | 模糊数据需求 | 拆解、清洗、建模 | 结构化数据集 |
| 课堂 | 学习目标 | 课程设计、授课 | 学员能力提升 |
| 咨询 | 业务问题 | 调研、诊断、方案 | 可执行建议 |
| 云服务 | 基础设施需求 | 部署、配置、运维 | 稳定运行环境 |

无论哪一项业务，Build in Public 加速的都是同一个东西：**从接收到交付再回到接收的循环速率**。

## 闭环速度决定竞争优势

当你的整个建设过程是公开的，你的每一圈循环都同时服务于三个目的：

1. **交付** — 完成当前业务需求
2. **记录** — 形成结构化的流程档案
3. **传播** — 让潜在客户和执行方看到你的工作方式

传统公司在这三者之间切换是有成本的：交付结束后要专门花时间写案例、整理复盘、走内部审批才能对外传播。在 Build in Public 模式下，交付本身就包含了记录和传播——你在做的时候别人就看到了，你做完了别人就知道你做到了。

这就是循环速率优势的来源：**别人一圈做完三件事，你一圈同时做完三件事。**

## Build in Public 本身就是护城河

> Build in Public 的核心不是开放，是速度。当验证和记录和传播同步发生时，你的对手不仅追不上你的信息，也追不上你的信息处理速度。
