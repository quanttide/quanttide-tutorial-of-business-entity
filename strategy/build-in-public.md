# Build in Public

## 什么是 Build in Public

"建在公开"不是"把手册公开"或"开源代码"。手册公开和开源是结果，不是策略。

Build in Public 是指在建设过程中，实时分享：

- 当前在做什么、为什么做
- 遇到了什么问题、怎么解决的
- 哪些尝试失败了、学到了什么
- 真实的数据——包括不好看的数据

GitLab 和 PostHog 是这一策略的代表。GitLab 把产品路线图和公司手册全部公开，任何决策的过程和理由都可在 handbook 中溯源。PostHog 实时公开 ARR、增长数据、定价调整的决策逻辑。YC 参投时看过他们的数据才放心，但那些数据早就被所有竞争对手看到了。

## 为什么 Build in Public 是护城河

传统直觉：公开信息会削弱护城河。

实际机制：**当信息本身不再是稀缺品时，它从资产变成了基础设施。而真正的护城河，从"怎么藏着信息"变成了"用什么速度消化这些信息"。**

可以画一条曲线：

```
信息开放程度
    ↑
    |          你在这里
    |          （GitLab/PostHog/量潮）
    |
    |      传统信任型公司
    |      （公开年报，隐藏过程）
    |
    |  传统保密型公司
    |  （配方/策略/数据全部保密）
    +------------------------→ 护城河强度
```

量潮和组织内的大量文档、战略、日记、数据全部公开，但我们不靠信息差建立护城河。我们的护城河来自**更快的闭环速度**。

## Build in Public × 数据服务：闭环速度

数据服务业务的核心能力不是技术，是**需求拆解和交付验证的闭环速度**：

```
需求 → 文档化 → 拆解 → 执行 → 验收 → 记录 → 需求...
         ↑                           ↑
       公开规格                    公开验收标准
         ↓                           ↓
        让客户验证                  比外部团队更快记录
```

每一次闭环都是一次数据结构化的机会。结果越公开：

1. **客户越容易判断我们是否适合他们** — 降低销售成本
2. **潜在执行方越清楚我们要求的交付水平** — 提高供应商匹配效率
3. **我们自己越过早发现流程中的损耗** — journal 里说的"内部数据采集不到位"，公开后外部视角自动补位

闭环速度决定竞争优势。当你的整个建设过程是公开的，你的每一圈闭环都同时服务于三个目的：交付、记录、传播。传统公司在交付结束后还要花时间写案例、整理复盘、走内部沟通，这些步骤在 Build in Public 模式下是交付本身的一部分。

## Build in Public 本身就是护城河

> Build in Public 的核心不是开放，是速度。当验证和记录和传播同步发生时，你的对手不仅追不上你的信息，也追不上你的信息处理速度。
