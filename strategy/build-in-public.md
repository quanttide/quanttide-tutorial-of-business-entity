# Build in Public

## 我们为什么建在公开

你可能注意到，我们所有的仓库都是公开的。从章程、手册到战略意图、业务日记，甚至你正在读的这份教程，都在 GitHub 上谁都能看。

这不是因为我们不小心把权限设错了。这是我们的工作方式。

我们说的 Build in Public，不是"把手册公开"或"开源代码"——那些是结果，不是策略。Build in Public 指的是在建设过程中，实时分享：

- 我们当前在做什么、为什么做
- 遇到了什么问题、怎么解决的
- 哪些尝试失败了、学到了什么
- 真实的数据——包括不好看的数据

## 为什么这样做

你可能觉得公开信息会削弱竞争力。传统的商业直觉确实是这么说的——配方保密、策略保密、数据保密，公开等于放弃护城河。

但如果这条路是唯一的答案，GitLab 和 PostHog 就不会存在了。GitLab 把产品路线图和公司手册全部公开，任何决策的过程和理由都可在 handbook 中溯源。PostHog 实时公开 ARR、增长数据、定价调整的决策逻辑。YC 参投时看过他们的数据才放心——但那些数据早就被所有竞争对手看到了。

这些公司证明了另一条路走得通：**当信息本身不再是稀缺品时，它从资产变成了基础设施。而真正的护城河，从"怎么藏着信息"变成了"用什么速度消化这些信息"。**

可以画一条曲线：

```
信息开放程度
    ↑
    |          我们在这里
    |          （GitLab/PostHog/量潮）
    |
    |      传统信任型公司
    |      （公开年报，隐藏过程）
    |
    |  传统保密型公司
    |  （配方/策略/数据全部保密）
    +------------------------→ 护城河强度
```

我们不靠信息差建立护城河。我们的护城河来自**更快的闭环速度**。

## 闭环速度：所有业务的共同结构

你可能在不同的业务线——数据、课堂、咨询、云——但所有业务的运转都可以抽象为同一个核心循环：

```
接收 → 处理 → 交付 → 记录 → 接收...
  ↑                    ↑
 公开输入规范          公开输出标准
  ↓                    ↓
 让对方验证           比对手更快归档
```

每一次循环都是一次信息结构化的机会。结果越公开：

1. **客户越容易判断是否与我们合作** — 降低我们的销售成本
2. **执行方越清楚我们要求的交付水平** — 提高匹配效率
3. **我们越早发现流程损耗** — 你会在 journal 里看到"内部数据采集不到位"这个老问题，公开后外部视角自动补位

不同业务的区别只在于循环里跑的是什么：

| 业务 | 接收（输入） | 处理 | 交付（输出） |
|------|------------|------|------------|
| 数据服务 | 模糊数据需求 | 拆解、清洗、建模 | 结构化数据集 |
| 课堂 | 学习目标 | 课程设计、授课 | 学员能力提升 |
| 咨询 | 业务问题 | 调研、诊断、方案 | 可执行建议 |
| 云服务 | 基础设施需求 | 部署、配置、运维 | 稳定运行环境 |

无论你在哪个业务线，Build in Public 加速的都是同一个东西：**从接收到交付再回到接收的循环速率。**

## 这给你的工作带来什么

当我们的整个建设过程是公开的，你的每一圈循环都同时服务于三个目的：

1. **交付** — 完成当前业务需求
2. **记录** — 形成结构化的流程档案（你的工作本身就在积累我们的知识库）
3. **传播** — 让潜在客户和执行方看到我们的工作方式（你不需要额外花时间写案例）

传统公司在这三者之间切换是有成本的：交付结束后要专门花时间写案例、整理复盘、走内部审批才能对外传播。我们不需要——你在做的时候别人就看到了，你做完了别人就知道你做到了。

这就是我们的循环速率优势：**别人一圈做完三件事，我们一圈同时做完三件事。**

## 护城河的最终形态

> Build in Public 的核心不是开放，是速度。当验证和记录和传播同步发生时，对手不仅追不上我们的信息，也追不上我们的信息处理速度。
