# 技术序列考核规则

## 一、考核的核心思想

我们的笔试，不是考知识，也不是考熟练度，而是考一件事：候选人的思维结构，是否和我们同频。

### 1.1 什么是"思维结构同频"

我们项目的核心方法论，是用"宪法"和"判例"来理解和设计系统：

· 看到具体实现时，习惯性地想：这里面哪些是通用的？哪些是绑定了具体平台的？
· 接入新场景时，本能地先去抽象共性，再做具体适配，而不是平铺逻辑。
· 对"硬编码假设"有洁癖——即使代码能跑，也会觉得它不够干净，会主动去想怎么拆。

如果一个人天然地用这种方式思考，他不需要我们反复解释"宪法"和"判例"是什么——他看完代码，自己就能感觉到哪里应该抽象、哪里是硬编码。

我们要找的，就是这种思维结构。

### 1.2 为什么不用传统面试

传统面试考算法、考八股文、考项目经验，但这些都无法回答一个关键问题：他能不能用我们的方法，解决他没遇到过的问题？

我们的项目在快速迭代，工具会变、平台会变、判例会变。唯一不变的是我们拆解问题的方式。所以考核必须直接检验这种拆解方式，而不是间接推测。

### 1.3 为什么用笔试而非实时面试

实时面试有时间压力，容易考成"快速反应"而不是"深度思考"。我们的项目需要深度思考——设计一个抽象模型，需要反复推敲、推倒重来、沉淀后再表达。

笔试给 2-3 天，模拟的是真实工作节奏：拿到问题，自己研究，产出方案，书面表达。这本身就是我们日常协作的方式。

## 二、试题设计原则

### 2.1 题目可以换，但结构不变

不管具体考什么技术栈、什么平台，试题的三层结构是固定的：

· 第一题：识别硬编码假设 → 考解耦直觉
· 第二题：设计抽象模型 → 考宪法思维
· 第三题：设计自举机制 → 考工程哲学

这三层不是难度递进，而是思维深度递进。第一题看他对现状的敏感度，第二题看他能不能从具体到抽象，第三题看他能不能从抽象到愿景。

### 2.2 题目必须来自真实项目

不要出道具题。用我们自己的开源项目，用我们正在解决的现实问题。

这样做的原因：

· 候选人面对的是真实代码，不是人工构造的练习题。这能测出他真正的工程阅读能力。
· 如果他对了，他产生的方案对我们有实际价值。考核本身就是一次协作。
· 他能通过题目感受到我们在做什么、我们怎么思考。题目本身就是一种信息传递。

### 2.3 不给标准答案

宪法模型没有唯一正确解。核心属性选哪几个、舍弃哪些、扩展点怎么设计——这些都有取舍。我们要看的是他如何做取舍，以及他能否清晰解释取舍的理由。

所以评分标准不是"对不对"，而是"决策是否有清晰的逻辑，假设是否被显式化"。

## 三、评估维度详解

### 3.1 对项目核心概念的理解深度

看什么：他是否理解了"宪法"和"判例"的区别，以及为什么要有这个区别。

合格信号：

· 不会把"支持多个平台"简单理解为"多写几个 if-else"
· 在设计模型时，能区分"属于所有平台的共性"和"属于某个平台的特性"
· 能把我们现有的 GitHub 实现看作一个"判例"，而不是"唯一实现"

不合格信号：

· 认为抽象是过度设计，"直接适配就行了"
· 把判例当宪法用——比如认为所有代码仓库都应该有 GitHub 那样的 Organization 概念

### 3.2 抽象建模能力

看什么：他能不能从具体中提炼普遍规则，能不能干净地定义边界。

合格信号：

· 定义的属性和方法足够通用，能覆盖已知案例
· 有明确的舍弃——能说清楚"为什么这个属性不应该放在抽象层"
· 模型有清晰的扩展点，知道哪里该留接口、哪里该写死

不合格信号：

· 模型过于具体，实际只是把 GitHub 的 API 重新命名了一遍
· 模型过于空泛，没有任何约束力，什么都能往里塞
· 没有取舍意识，"这个模型里什么都有"

### 3.3 工程判断力

看什么：他做设计决策时，有没有清晰的权衡逻辑。

合格信号：

· 每个决策都有"为什么"的解释
· 能主动说明自己做的假设——"这里我没想清楚，暂时假设 X"
· 能讨论方案的局限性和风险

不合格信号：

· 决策没有解释，"我觉得这样比较好"
· 方案完美主义，看不出任何取舍和边界
· 不考虑失败场景，默认一切都会按预期运行

### 3.4 沟通质量

看什么：他的工作蓝图，是否能让别人高效地理解他的思路。

合格信号：

· 文档结构清晰，有关键的架构图或流程图
· 假设被显式声明，不确定的地方被标记出来
· 能用简洁的语言描述复杂的设计

不合格信号：

· 长篇大论但没有重点
· 图表缺失或粗糙到无法理解
· 不标注假设，让人不知道哪里是确定的、哪里是他猜的

## 四、操作步骤

### 4.1 发题前

1. 确认当前项目的"硬编码热点"：你自己先过一遍代码，心里有数哪些地方是硬编码的。这样你看候选人分析时，能判断他是真正找到了问题，还是在泛泛而谈。
2. 确认当前宪法模型的"争议点"：哪些抽象维度我们自己也在纠结？这能帮你在评估第二题时，判断候选人是否触及了真正深水区的问题。
3. 准备项目入口信息：确保候选人能顺畅拉取代码、看懂 README、找到关键入口。如果他卡在环境搭建上，那不是我们要考察的东西。

### 4.2 考核期间

1. 完全异步，不催促：给了 2-3 天就让他用完。不要中途问进度。自主管理时间是考核的一部分。
2. 对问题的回应保持克制：如果他通过 Issue 提问，只回答最必要的澄清。不要帮他思考，不要暗示方向。如果他问"这里应该抽象成几个接口"，回答"这是你的设计决策，请你在工作蓝图中说明你的取舍理由"。
3. 观察他的提问质量：他问了什么，本身就是重要的考核信号。问"这个函数的参数是什么"是平庸的，问"你们为什么选择这样定义而不是那样定义"是好的。

### 4.3 评估时

1. 先整体后细节：先读完整个工作蓝图，对思维深度有一个整体判断，再逐项打分。不要被某个具体错误或亮点带偏整体印象。
2. 关注决策逻辑而非结论：他的模型设计和你预期不一样，不等于他错。关键是看他怎么推出来的。如果逻辑自洽、假设清晰，即使结论不同，也是合格甚至优秀的。
3. 用四个评估维度分别打分，然后综合定级：

等级 特征
S 三题都出色，抽象模型干净且可扩展，第三题有启发性思路，能在设计中反馈我们对项目的启发
A 前两题出色，模型设计合理有取舍，第三题方案常规但逻辑自洽
B 第一题能识别问题但方案偏平铺，第二题理解抽象必要性但模型粗糙，需要引导
C 不理解为什么要抽象，方案停留在"多加几个 if"的层面

### 4.4 评估后

1. 给每个候选人反馈：无论是否通过，简短说明评估结果。这是我们作为开源社区的尊重，也帮候选人理解我们的标准。
2. 如果通过，用他的工作蓝图作为入职后的第一个工作起点：他已经在这份蓝图里表达了对项目的理解和设计思路，入职后可以直接从这个蓝图开始落地。

## 五、常见问题

Q：如果候选人完全不理解"宪法/判例"这两个概念怎么办？
A：背景说明里已经解释了这两个概念。如果他看完代码、读完说明，仍然不理解为什么要抽象，那就是思维结构不匹配。这不是能力问题，是方法论的契合度问题。

Q：候选人的方案和我们的现有设计完全不一样，怎么评估？
A：看他的逻辑和取舍。如果他的方案逻辑自洽，并且清晰说明了为什么这样选择，即使和我们不一样，也可能是优秀的设计。我们的设计不是唯一正确答案。

Q：笔试时间太长，候选人会不会放弃？
A：会。这正是筛选的一部分。愿意花 2-3 天认真研究我们项目的人，本身就对这个方向有足够的兴趣。那些觉得"太麻烦了"的人，大概率也不会享受我们的工作方式。

—

这份教程的核心就一句话：我们不考知识，我们考思维结构。题目是载体，宪法和判例是方法，自举是最终方向。
