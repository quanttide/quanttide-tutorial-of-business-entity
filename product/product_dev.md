# 数据驱动的产品研发

## 意图

建立从实际业务数据到业务平台的标准化产品研发流程，解决传统研发中文档与代码割裂的核心矛盾。

## 内在逻辑

### 核心矛盾：文档与代码的割裂

传统研发中，文档和代码是两条线：

```
文档（需求/设计/手册）→ 人读 → 人编码 → 代码
                                                ↓
                                      文档又得人手动更新
```

文档是"参考"而非"契约"。业务知识在文档里腐烂，代码里硬编码的逻辑也随之腐烂。两者靠人来同步——人是最薄弱的环节。

### 解决思路：用半结构化数据桥接两层

```
人写的自然语言笔记
        ↓ 结构化提取
   JSON/YAML 契约
        ↓ 各端读取
   CLI / Studio / Site 等实现
        ↓ 产生新数据
   回写到档案
```

关键变化：**文档不再直接写给"人看"或"代码用"，而是先提炼成机器可读的中间层。**

- 人继续写自然语言（草稿、笔记、档案）
- 多一步：把知识中可结构化的部分抽成 JSON/YAML
- 各端代码读 JSON/YAML，不读 Markdown

知识的变动只需更新 JSON/YAML，各端自然同步。

### 三种知识形态的分离

这套模式背后隐含了对知识三种形态的区分：

| 形态 | 载体 | 消费者 | 例子 |
|------|------|--------|------|
| **叙事知识** | Markdown 文档 | 人类 | 方法描述、需求说明 |
| **结构化知识** | JSON/YAML | 程序/CLI/Studio | 资产类型定义、分类体系 |
| **可编程知识** | 代码逻辑 | 机器引擎 | CLI/Studio 的实现代码 |

传统研发的误区：三种知识混在一起。需求文档里有分类，代码里也硬编码了同样的分类，改一处漏一处。

正确做法：**把结构化知识从叙事和代码中显式提取出来，作为独立的契约层。三层各自维护，互不干扰。**

### 知识增殖闭环

```
档案 → 契约 → 代码 → 新数据 → 档案
```

这不是单向流水线，而是知识增殖循环：

1. 档案里已有的知识被结构化为契约
2. 契约驱动各端开发
3. 开发过程中发现新的分类、新的资产、新的关系
4. 这些新发现被回写到档案
5. 档案更丰富 → 可以提取更多契约 → ...

每轮循环，档案都变得更精炼，而非传统文档越写越乱。

## 流程

1. 清洗和精炼工作档案的业务数据并进行知识建模。
2. 使用半结构化数据作为契约驱动各端独立开发。
3. 产生新数据进入工作档案。

## 验收

1. 业务数据是否更精炼和具备直接使用价值。
2. 业务代码是否具备可复用价值。
