# 考核方法

量潮考核分**市场序列**与**技术序列**。

两者共享同一套设计哲学——不考知识/技巧，考判断力和思维结构。题目是载体，不是目的。评分标准不是"对不对"，而是"决策逻辑是否清晰"。

---

## 市场序列

我们不考文案技巧，不考排版审美，也不考"你做过多少个爆款"。

我们只考一件事：候选人能不能把内部经验，转化为外部用户想看的商业内容。

这不是写爆款文案，这是做认知翻译。

### 试题

三层结构，递进设计：

1. **拆解一篇高转化内容** → 考分析能力和方法论提炼
2. **改写一篇内部经验文章** → 考核心工作能力（内部到外部的翻译）
3. **简要规划系列内容** → 考延伸能力和系统思维

题目必须来自真实材料，不给标准答案，看取舍。

### 评估维度

一、**改写能力**。看候选人能否将内部经验分享，改写成面向外部读者的商业内容。

合格：保留了原文最有冲击力的核心素材，调整了叙事视角（从"我做了什么"转为"这和你有什么关系"），删减了内部语境，在自然节点植入转化。

不合格：改写稿和原文几乎没有区别，或者把原文改成了满篇广告。

二、**叙事意识**。看候选人是否理解"真实挣扎故事"的叙事逻辑。

合格：文章有明确的"事故"或"冲突"作为锚点，展示了真实的挣扎过程，读起来真实。

不合格：成功学叙事——"我们做了什么，效果多么好"，没有冲突，没有转折。

三、**转化设计**。看候选人能否设计自然的转化路径。

合格：课程信息出现的位置水到渠成，读者看到那个地方会觉得"对，我想了解更多"。

不合格：结尾突然硬推销，或者全文多处重复出现课程信息。

四、**用户视角**。看候选人是否始终在想"读者读到这会怎么想"。

合格：开头能让人产生代入感，节奏符合认知曲线，用语通俗。

不合格：文章是写给同行看的，大量使用读者不理解的概念。

### 定级

| 等级 | 特征 |
|------|------|
| S | 改写稿可直接发布，转化自然，改写说明清晰 |
| A | 质量好，转化合理，知道自己在做什么决策 |
| B | 方向对但落地有偏差，需要带教 |
| C | 和原文几乎一样，或变成纯粹广告 |

核心就一句话：我们不考文笔，我们考判断力。题目是改写，考的是认知翻译。

## 技术序列

我们不考知识，也不考熟练度。

我们只考一件事：候选人的思维结构，是否和我们同频。

我们的核心方法论，是用"宪法"和"判例"来理解和设计系统。看到具体实现时，习惯性地想哪些是通用的、哪些是绑定了平台的；接入新场景时，本能地先抽象共性再做适配。

我们要找的，就是这种思维结构。

### 试题

三层结构，思维深度递进：

1. **识别硬编码假设** → 考解耦直觉
2. **设计抽象模型** → 考宪法思维
3. **设计自举机制** → 考工程哲学

题目必须来自真实项目，不给标准答案——看取舍。

### 评估维度

一、**概念理解**。看候选人是否理解了"宪法"和"判例"的区别。

合格：不会把"支持多个平台"简单理解为多写几个 if-else，能把现有实现看作判例而非唯一实现。

不合格：认为抽象是过度设计，或者把判例当宪法用。

二、**抽象建模能力**。看候选人能否从具体中提炼普遍规则。

合格：定义的属性和方法足够通用，有明确的舍弃，模型有清晰的扩展点。

不合格：模型过于具体（只是重命名了现有 API），或过于空泛（什么都能往里塞）。

三、**工程判断力**。看候选人做设计决策时是否有清晰的权衡逻辑。

合格：每个决策都有"为什么"的解释，能主动声明假设，能讨论局限性。

不合格：决策没有解释，方案完美主义看不出取舍，不考虑失败场景。

四、**沟通质量**。看候选人的工作蓝图能否让别人高效理解。

合格：文档结构清晰，有关键图表，假设被显式声明。

不合格：长篇大论没有重点，图表缺失或粗糙。

### 定级

| 等级 | 特征 |
|------|------|
| S | 三题都出色，模型干净可扩展，第三题有启发性 |
| A | 前两题出色，模型合理有取舍，第三题逻辑自洽 |
| B | 能识别问题但方案偏平铺，模型粗糙需引导 |
| C | 不理解为何要抽象 |

核心就一句话：我们不考知识，我们考思维结构。题目是载体，宪法和判例是方法，自举是最终方向。

---

## 操作规范

**发题前：** 面试官先熟悉材料，确保候选人能顺畅获取所需信息。

**考核期间：** 2-3 天完全异步，不催促。只回答事实性问题，不暗示方向。候选人提什么问题，本身就是信号。

**评估后：** 无论是否通过，简短反馈。通过者的产出直接作为入职后第一个工作起点。
