# 开源协议选择策略

## 协议矩阵

| 范围 | 协议 | 分类 |
|---|---|---|
| `LICENSE`（根仓库） | CC-BY-4.0 | 文档 |
| `docs/`（15 个子模块） | CC-BY-4.0 | 文档 |
| `apps/`（6 个子模块） | Apache 2.0 | 软件 |

## 选择原则

1. **内容 vs 代码分治**：文档用 CC 系列，软件用 Apache 系列
2. **全部宽松（Permissive）**：不含传染性条款，允许闭源和商用
3. **全部允许商用**：降低客户/合作方采用门槛
4. **全部要求署名**：保留传播链中的品牌标识

## 文档协议选型

| 协议 | 是否采用 | 理由 |
|---|---|---|
| CC0 | 不采用 | 放弃全部权利，无法要求署名，品牌不可追溯 |
| **CC-BY** | **采用** | 最宽松的署名协议，要求来源标示，允许商用和改编 |
| CC-BY-SA | 不采用 | 相同方式共享有传染性，阻碍商业采用 |
| CC-BY-NC | 不采用 | 限制商用，与"让市场接受标准"矛盾 |

## 软件协议选型

| 协议 | 是否采用 | 理由 |
|---|---|---|
| MIT | 不采用 | 缺少专利授权条款，对贡献者保护不足 |
| **Apache 2.0** | **采用** | 含专利授权，保护贡献者和使用者，宽松且业界主流 |
| GPL | 不采用 | 强传染性，限制闭源商业使用 |

## 为什么要求署名

署名权是影响力的放大器。每次引用携带"量潮科技"品牌标识，形成网络效应；防止第三方抹去来源声称自有；为第二大脑外部参与通道（提 issue、交 PR、招募人才）提供可溯源的入口。
